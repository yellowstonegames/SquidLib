007/**
008 * A very-high-quality StatefulRandomness that is the fastest 64-bit generator in this library that passes statistical
009 * tests and is equidistributed. Has 64 bits of state and natively outputs 64 bits at a time, changing the state with a
010 * basic linear congruential generator (it is simply {@code state = state * 1103515245 + 1}). Starting with that LCG's
011 * output, it xorshifts that long, multiplies by a very large negative long, then returns another xorshift. For
012 * whatever reason, the output of this simple function passes all 32TB of PractRand with no anomalies, meaning its
013 * statistical quality is excellent. As mentioned earlier, this is the fastest high-quality generator here other than
014 * {@link ThrustAltRNG}. Unlike ThrustAltRNG, this can produce all long values as output; ThrustAltRNG bunches some
015 * outputs and makes producing them more likely while others can't be produced at all. Notably, this generator is faster
016 * than {@link LightRNG} while keeping the same or higher quality, and also faster than {@link XoRoRNG} while passing
017 * tests that XoRoRNG always or frequently fails, such as binary matrix rank tests. It is not the fastest option on GWT;
018 * if you intend to target GWT then {@link Lathe32RNG} or {@link GWTRNG} is recommended (they use the same algorithm,
019 * xoroshiro64++ as it has recently been termed, and it has almost identical period to LinnormRNG but avoids math on
020 * longs to increase performance on GWT).
021 * <br>
022 * This generator is a StatefulRandomness but not a SkippingRandomness, so it can't (efficiently) have the skip() method
023 * that LightRNG has. A method could be written to run the generator's state backwards, though, as well as to get the
024 * state from an output of {@link #nextLong()}. At least in theory, and supported in practice by how PCG-Random works,
025 * this type of generator could be used to make a generator with multiple non-overlapping streams by changing the linear
026 * congruential generator to add a different odd number instead of 1; any odd number should work, but optimizations can
027 * apply to adding 1 that aren't possible for other numbers, so that's a good default to use here. LinnormRNG has a
028 * static determine() method that uses a slightly different algorithm; it performs similar operations to
029 * {@link LightRNG#determine(long)}. That involves 3 multiplications by constants, 3 "xorshifts" that each use a XOR and
030 * a bitwise right shift on a local variable, and 3 temporary assignments; this determine() uses 3 multiplications by
031 * constants, only 2 xorshifts, only 2 temporary assignments, and one XOR by a constant. Like LightRNG's determine(),
032 * LinnormRNG.determine() can take any long as input and has all longs as possible outputs. Unlike LightRNG's
033 * determine(), an input of 0 does not produce a result of 0.
034 * <br>
035 * The name comes from LINear congruential generator this uses to change it state, while the rest is a NORMal
036 * SplitMix64-like generator. "Linnorm" is a Norwegian name for a kind of dragon, as well. 
037 * <br>
038 * Written May 9, 2018 by Tommy Ettinger.
039 * @author Tommy Ettinger
040 */
041public final class LinnormRNG implements RandomnessSource, StatefulRandomness, Serializable {

047    /**
048     * Constructor that seeds the generator with two calls to Math.random.
049     */
050    public LinnormRNG() {
051        this((long) ((Math.random() - 0.5) * 0x10000000000000L)
052                ^ (long) (((Math.random() - 0.5) * 2.0) * 0x8000000000000000L));
053    }

055    /**
056     * Constructor that uses the given seed as the state without changes; all long seeds are acceptable.
057     * @param seed any long, will be used exactly

063    /**
064     * Constructor that hashes seed with {@link CrossHash#hash64(CharSequence)} and uses the result as the state.
065     * @param seed any CharSequence, such as a String or StringBuilder; should probably not be null (it might work?)

071    @Override
072    public final int next(int bits)
073    {
074        long z = (state = state * 0x41C64E6DL + 1L);
075        z = (z ^ z >>> 27) * 0xAEF17502108EF2D9L;
076        return (int)(z ^ z >>> 25) >>> (32 - bits);
077    }

079    /**
080     * Can return any long, positive or negative, of any size permissible in a 64-bit signed integer.
081     *
082     * @return any long, all 64 bits are random
083     */
084    @Override
085    public final long nextLong() {
086        long z = (state = state * 0x41C64E6DL + 1L);
087        z = (z ^ z >>> 27) * 0xAEF17502108EF2D9L;
088        return (z ^ z >>> 25);
089    }

096     * @return a copy of this LinnormRNG

108    public final int nextInt() {
109        long z = (state = state * 0x41C64E6DL + 1L);

111        return (int)(z ^ z >>> 25);

114    /**
115     * Exclusive on the outer bound.  The inner bound is 0.
116     * The bound can be negative, which makes this produce either a negative int or 0.
117     *
118     * @param bound the upper bound; should be positive
119     * @return a random int between 0 (inclusive) and bound (exclusive)
120     */
121    public final int nextInt(final int bound) {
122        long z = (state = state * 0x41C64E6DL + 1L);
123        z = (z ^ z >>> 27) * 0xAEF17502108EF2D9L;
124        return (int)((bound * ((z ^ z >>> 25) & 0xFFFFFFFFL)) >> 32);
125    }

127    /**
128     * Inclusive inner, exclusive outer.
129     *
130     * @param inner the inner bound, inclusive, can be positive or negative
131     * @param outer the outer bound, exclusive, can be positive or negative, usually greater than inner
132     * @return a random int between inner (inclusive) and outer (exclusive)
133     */
134    public final int nextInt(final int inner, final int outer) {
135        return inner + nextInt(outer - inner);
136    }

138    /**
139     * Exclusive on bound (which may be positive or negative), with an inner bound of 0.
140     * If bound is negative this returns a negative long; if bound is positive this returns a positive long. The bound
141     * can even be 0, which will cause this to return 0L every time. This uses a biased technique to get numbers from
142     * large ranges, but the amount of bias is incredibly small (expected to be under 1/1000 if enough random ranged
143     * numbers are requested, which is about the same as an unbiased method that was also considered). It may have
144     * noticeable bias if the LinnormRNG's period is exhausted by only calls to this method, which would take months on
145     * 2018-era consumer hardware. Unlike all unbiased methods, this advances the state by an equivalent to exactly one
146     * call to {@link #nextLong()}, where rejection sampling would sometimes advance by one call, but other times by
147     * arbitrarily many more.
148     * <br>
149     * Credit for this method goes to <a href="https://oroboro.com/large-random-in-range/">Rafael Baptista's blog</a>,
150     * with some adaptation for signed long values and a 64-bit generator. This method is drastically faster than the
151     * previous implementation when the bound varies often (roughly 4x faster, possibly more). It also always gets at
152     * most one random number, so it advances the state as much as {@link #nextInt(int)} or {@link #nextLong()}.
153     * @param bound the outer exclusive bound; can be positive or negative
154     * @return a random long between 0 (inclusive) and bound (exclusive)
155     */
156    public long nextLong(long bound) {
157        long rand = (state = state * 0x41C64E6DL + 1L);
158        rand = (rand ^ rand >>> 27) * 0xAEF17502108EF2D9L;
159        rand ^= rand >>> 25;
160        final long randLow = rand & 0xFFFFFFFFL;
161        final long boundLow = bound & 0xFFFFFFFFL;
162        rand >>>= 32;
163        bound >>= 32;
164        final long z = (randLow * boundLow >> 32);
165        final long t = rand * boundLow + z;
166        return rand * bound + (t >> 32) + ((t & 0xFFFFFFFFL) + randLow * bound >> 32) - (z >> 63);
167    }
168    /**
169     * Inclusive inner, exclusive outer; lower and upper can be positive or negative and there's no requirement for one
170     * to be greater than or less than the other.
171     *
172     * @param lower the lower bound, inclusive, can be positive or negative
173     * @param upper the upper bound, exclusive, can be positive or negative
174     * @return a random long that may be equal to lower and will otherwise be between lower and upper
175     */
176    public final long nextLong(final long lower, final long upper) {
177        return lower + nextLong(upper - lower);
178    }

215    /**
216     * Gets a random value, true or false.
217     * Calls nextLong() once.
218     *
219     * @return a random true or false value.
220     */
221    public final boolean nextBoolean() {
222        long z = (state = state * 0x41C64E6DL + 1L);
223        return ((z ^ z >>> 27) * 0xAEF17502108EF2D9L) < 0;
224    }

226    /**
227     * Given a byte array as a parameter, this will fill the array with random bytes (modifying it
228     * in-place). Calls nextLong() {@code Math.ceil(bytes.length / 8.0)} times.
229     *
230     * @param bytes a byte array that will have its contents overwritten with random bytes.
231     */
232    public final void nextBytes(final byte[] bytes) {
233        int i = bytes.length, n;
234        while (i != 0) {
235            n = Math.min(i, 8);
236            for (long bits = nextLong(); n-- != 0; bits >>>= 8) bytes[--i] = (byte) bits;
237        }
238    }

277    /**
278     * Static randomizing method that takes its state as a parameter; state is expected to change between calls to this.
279     * It is recommended that you use {@code LinnormRNG.determine(++state)} or {@code LinnormRNG.determine(--state)} to
280     * produce a sequence of different numbers, but you can also use {@code LinnormRNG.determine(state += 12345L)} or
281     * any odd-number increment. All longs are accepted by this method, and all longs can be produced; unlike several
282     * other classes' determine() methods, passing 0 here does not return 0.
283     * @param state any long; subsequent calls should change by an odd number, such as with {@code ++state}
284     * @return any long
285     */
286    public static long determine(long state)
287    {
288        return (state = ((state = (((state * 0x632BE59BD9B4E019L) ^ 0x9E3779B97F4A7C15L) * 0xC6BC279692B5CC83L)) ^ state >>> 27) * 0xAEF17502108EF2D9L) ^ state >>> 25;
289    }

291    /**
292     * Static randomizing method that takes its state as a parameter and limits output to an int between 0 (inclusive)
293     * and bound (exclusive); state is expected to change between calls to this. It is recommended that you use
294     * {@code LinnormRNG.determineBounded(++state, bound)} or {@code LinnormRNG.determineBounded(--state, bound)} to
295     * produce a sequence of different numbers, but you can also use
296     * {@code LinnormRNG.determineBounded(state += 12345L, bound)} or any odd-number increment. All longs are accepted
297     * by this method, but not all ints between 0 and bound are guaranteed to be produced with equal likelihood (for any
298     * odd-number values for bound, this isn't possible for most generators). The bound can be negative.
299     * @param state any long; subsequent calls should change by an odd number, such as with {@code ++state}
300     * @param bound the outer exclusive bound, as an int
301     * @return an int between 0 (inclusive) and bound (exclusive)
302     */
303    public static int determineBounded(long state, final int bound)
304    {
305        return (int)((bound * (((state = ((state = (((state * 0x632BE59BD9B4E019L) ^ 0x9E3779B97F4A7C15L) * 0xC6BC279692B5CC83L)) ^ state >>> 27) * 0xAEF17502108EF2D9L) ^ state >>> 25) & 0x7FFFFFFFL)) >> 31);
306    }

309     * Returns a random float that is deterministic based on state; if state is the same on two calls to this, this will
310     * return the same float. This is expected to be called with a changing variable, e.g. {@code determine(++state)},
311     * where the increment for state should be odd but otherwise doesn't really matter. This multiplies state by
312     * {@code 0x632BE59BD9B4E019L} within this method, so using a small increment won't be much different from using a
313     * very large one, as long as it is odd. The period is 2 to the 64 if you increment or decrement by 1, but there are
314     * less than 2 to the 30 possible floats between 0 and 1.

317     *              generate numbers in reverse order

319     */
