009/**
010 * Simple hashing functions that we can rely on staying the same cross-platform.
011 * The static methods of this class (not its inner classes) use the Fowler/Noll/Vo
012 * Hash (FNV-1a) algorithm, which is public domain. The hashes this returns are always
013 * 0 when given null to hash. Arrays with identical elements of identical types will
014 * hash identically. Arrays with identical numerical values but different types will
015 * hash differently (only for FNV-1a, not the inner classes). FNV-1a may be somewhat
016 * slow if you are running many hashes or hashing very large data; in that case you
017 * should consider one of the inner classes, Lightning or Storm.
018 * <br>
019 * There are two static inner classes in CrossHash, Lightning and Storm, that provide different
020 * hashing properties, as well as the inner IHasher interface.
021 * <br>
022 * IHasher values are provided as static fields, and use FNV-1a to hash a specific type or fall
023 * back to Object.hashCode if given an object with the wrong type. IHasher values are optional
024 * parts of OrderedMap and OrderedSet, and allow arrays to be used as keys in those collections
025 * while keeping hashing by value instead of the normal hashing by reference for arrays. You
026 * probably won't ever need to make a class that implements IHasher yourself.
027 * <br>
028 * The inner classes provide alternate, faster hashing algorithms. Lightning has no theoretical
029 * basis or grounds in any reason other than empirical testing for why it does what it does, and
030 * this seems to be in line with many widely-used hashes (see: The Art of Hashing,
031 * http://eternallyconfuzzled.com/tuts/algorithms/jsw_tut_hashing.aspx ). That said, it performs
032 * very well, just slightly behind Arrays.hashCode (18-20 ms instead of 15 ms for over a million
033 * hashes of 16-element long arrays, including overhead for generating them, while SipHash and
034 * FNV-1a take approximately 80 ms and 135-155 ms, respectively, for the same data).
035 * <br>
036 * Storm is a variant on Lightning with 64 bits for a salt-like modifier as a member variable,
037 * which can make 2 to the 64 individual hashing functions from one set of code. Storm has some
038 * properties of a cryptographic hash, but is not recommended it for that usage. It is, however
039 * ideal for situations that show up often in game development where end users may be able to see
040 * and possibly alter some information that you don't want changed (i.e. save data stored on a
041 * device or in the browser's LocalStorage). If you want a way to verify the data is what you
042 * think it is, you can store a hash, using one of the many-possible hash functions this can
043 * produce, somewhere else and verify that the saved data has the hash it did last time; if the
044 * exact hashing function isn't known (or exact functions aren't known) by a tampering user,
045 * then it is unlikely they can make the hash match even if they can edit it. Storm is slightly
046 * slower than Lightning, at about 25-26 ms for a million hashes (still including some RNG
047 * overhead) instead of Lightning's 18-20 ms, but should never be worse than twice as slow as
048 * Arrays.hashCode, and is still about three times faster than the similar SipHash that SquidLib
049 * previously had in this class.
050 * <br>
051 * All of the hashes used here have about the same rate of collisions on identical data
052 * (testing used Arrays.hashCode, FNV-1a, Lightning, Storm, and the now-removed SipHash), but
053 * Lightning (and Storm) seem to do better than the rest on collision rates more often than
054 * they do worse than all the others. Lightning has been changed frequently but is considered
055 * stable now, but it isn't being considered to replace the FNV-1a algorithm in CrossHash for
056 * compatibility reasons. It seems to meet all the criteria for a good hash function, though,
057 * including doing well with a visual test that shows issues in FNV-1a and especially
058 * Arrays.hashCode. Storm is still not necessarily in a final version; the commit that removed
059 * SipHash also changed Storm's algorithm to more heavily factor in the salt and allow that
060 * salt to alter any patterns present in the output, making similar data hash very differently
061 * when the salts are different.
062 * <br>
063 * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
064 * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
065 * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing SipHash)
066 * does approximately as well as Lightning, with no clear patterns visible (Sip has been removed from SquidLib
067 * because it needs a lot of code and is slower than Storm and especially Lightning). The idea is from a
068 * technical report on visual uses for hashing, http://www.clockandflame.com/media/Goulburn06.pdf .
069 * <ul>
070 * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
071 * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
072 * <li>(Former) CrossHash.Sip.hash(int[]): http://i.imgur.com/keSpIwm.png</li>
073 * <li>{@link CrossHash.Lightning#hash(int[])}: http://i.imgur.com/afGJ9cA.png</li>
074 * </ul>
075 * <br>
076 * Note: This class was formerly called StableHash, but since that refers to a specific
077 * category of hashing algorithm that this is not, and since the goal is to be cross-
078 * platform, the name was changed to CrossHash.
079 * Created by Tommy Ettinger on 1/16/2016.
080 *

088        if (data == null)
089            return 0;

091        for (int i = 0; i <= len; i++) {
092            o |= (data[i]) ? (1 << (i & 7)) : 0;
093            if ((i & 7) == 7 || i == len) {
094                h ^= o;
095                h *= 16777619;
096                o = 0;
097            }
098        }
099        return h;

603    private static boolean equalityHelper(Object[] left, Object[] right, IHasher inner)
604    {
605        if(left == right)
606            return true;
607        if((left == null) ^ (right == null))
608            return false;
609        for (int i = 0; i < left.length && i < right.length; i++) {
610            if(!inner.areEqual(left[i], right[i]))
611                return false;
612        }
613        return true;
614    }

824    /**
825     * A quick, simple hashing function that seems to have good results. Like LightRNG, it stores a state that
826     * it updates independently of the output, and this starts at a large prime. At each step, it takes the
827     * current item in the array being hashed, adds a large non-prime used in LightRNG's generation function
828     * (it's 2 to the 64, times the golden ratio phi, and truncated to a signed long), multiplies by a prime
829     * called the "state multiplier", adds the result to the state and stores it, multiplies the value of the
830     * state by another prime called the "output multiplier", then XORs the current result with that value
831     * before moving onto the next item in the array. A finalization step XORs the result with a complex value
832     * made by adding the state (left over from the previous step) to what was the output multiplier, adding
833     * the last known value for result to the phi-related constant from LightRNG, multiplying that pair, adding
834     * the initial state (which turns out to be unusually good for this, despite no particularly special numeric
835     * qualities other than being a probable prime) and then bitwise-rotating it left by a seemingly-random
836     * number drawn from the highest 6 bits of the state.
837     * <br>
838     * This all can be done very quickly; a million hashes of a million different 16-element long arrays can be
839     * computed in under 18-20 ms (in the benchmark, some amount of that is overhead from generating a new
840     * array with LongPeriodRNG, since the benchmark uses that RNG's state for data, and the default
841     * Arrays.hashCode implementation is only somewhat faster at under 16 ms). After several tries and tweaks
842     * to the constants this uses, it also gets remarkably few hash collisions. On the same 0x100000, or
843     * 1048576, RNG states for data, Lightning gets 110 collisions, the JDK Arrays.hashCode method gets 129
844     * collisions, Sip (implementing SipHash) gets 145 collisions, and CrossHash (using the FNV-1a algorithm)
845     * gets 135 collisions. Storm depends on the salt chosen, but with one initialized with the phi-based
846     * constant that shows up in LightRNG and here, Storm gets 116 collisions. Dispersion is not perfect, but
847     * at many bit sizes Lightning continues to have less collisions (it disperses better than the other hashes
848     * with several quantities of bits, at least on this test data). Lightning also does relatively well, though
849     * it isn't clearly ahead of the rest all the time, when hashing Strings, especially ones that use a larger
850     * set of letters, it seems (FakeLanguageGen was used to make test data, and languages that used more
851     * characters in their alphabet seemed to hash better with this than competing hashes for some reason).
852     * <br>
853     * There is certainly room for improvement with the specific numbers chosen; earlier versions used the
854     * state multiplier "Neely's number", which is a probable prime made by taking the radix-29 number
855     * "HARGNALLINSCLOPIOPEEPIO" (a reference to a very unusual TV show), truncating to 64 bits, and rotating
856     * right by 42 bits. This version uses "Neely's number" for an initial state and during finalization, and
857     * uses a different probable prime as the state multiplier, made with a similar process; it starts with the
858     * radix-36 number "EDSGERWDIJKSTRA", then does the same process but rotates right by 37 bits to obtain a
859     * different prime. This tweak seems to help with hash collisions. Extensive trial and error was used to
860     * find the current output multiplier, which has no real relationship to anything else but has exactly 32 of
861     * 64 bits set to 1, has 1 in the least and most significant bit indices (meaning it is negative and odd),
862     * and other than that seems to have better results on most inputs for mystifying reasons. Earlier versions
863     * applied a Gray code step to alter the output instead of a multiplier that heavily overflows to obfuscate
864     * state, but that had a predictable pattern for most of the inputs tried, which seemed less-than-ideal for
865     * a hash. Vitally, Lightning avoids predictable collisions that Arrays.hashCode has, like
866     * {@code Arrays.hashCode(new long[]{0})==Arrays.hashCode(new long[]{-1})}.
867     * <br>
868     * The output multiplier is 0xC6BC279692B5CC83L, the state multiplier is 0xD0E89D2D311E289FL, the number
869     * added to the state (from LightRNG and code derived from FastUtil, but obtained from the golden ratio
870     * phi) is 0x9E3779B97F4A7C15L, and the starting state ("Neely's Number") is 0x632BE59BD9B4E019L.
871     * <br>
872     * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
873     * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
874     * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
875     * SipHash) does approximately as well as Lightning, with no clear patterns visible (Sip has been removed
876     * from SquidLib because it needs a lot of code and is slower than Storm and especially Lightning). The
877     * idea is from a technical report on visual uses for hashing,
878     * http://www.clockandflame.com/media/Goulburn06.pdf .
879     * <ul>
880     * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
