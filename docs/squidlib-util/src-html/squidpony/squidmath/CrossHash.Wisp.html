010/**
011 * Simple hashing functions that we can rely on staying the same cross-platform.
012 * The static methods of this class (not its inner classes) use a custom algorithm
013 * designed for speed and general-purpose usability, but not cryptographic security;
014 * this algorithm is sometimes referred to as Hive, and several other algorithms are
015 * available in static inner classes, some of which have different goals, such as
016 * reduced likelihood of successfully reversing a hash, or just providing another
017 * choice. The hashes this returns are always 0 when given null to hash. Arrays with
018 * identical elements of identical types will hash identically. Arrays with identical
019 * numerical values but different types will sometimes hash differently. This class
020 * always provides 64-bit hashes via hash64() and 32-bit hashes via hash(), and some
021 * of the algorithms here may provide a hash32() method that matches older behavior
022 * and uses only 32-bit math. The hash64() and hash() methods, at least in Wisp and
023 * Mist, use 64-bit math even when producing 32-bit hashes, for GWT reasons. GWT
024 * doesn't have the same behavior as desktop and Android applications when using ints
025 * because it treats doubles mostly like ints, sometimes, due to it using JavaScript.
026 * If we use mainly longs, though, GWT emulates the longs with a more complex
027 * technique behind-the-scenes, that behaves the same on the web as it does on
028 * desktop or on a phone. Since CrossHash is supposed to be stable cross-platform,
029 * this is the way we need to go, despite it being slightly slower.
030 * <br>
031 * There are several static inner classes in CrossHash: Lightning, Falcon, and Mist,
032 * each providing different hashing properties, as well as the inner IHasher interface and a
033 * compatibility version of Wisp as a subclass. Older versions of SquidLib encouraged using a
034 * subclass because the non-nested-class methods used a lousy implementation of the FNV-1a algorithm,
035 * which was roughly 10x slower than the current methods in CrossHash and had poor correlation
036 * properties. In the current version, you probably will be fine with the default functions in
037 * CrossHash, using the Wisp algorithm. If you need a salt to alter the hash function,
038 * using one of a large family of such functions instead of a single function like Wisp, then Mist
039 * is a good choice. Lightning is mostly superseded by Wisp, but it can have better behavior on some
040 * collections regarding collisions; Falcon is meant to be a faster version of Lightning.
041 * <br>
042 * IHasher values are provided as static fields, and use Wisp to hash a specific type or fall
043 * back to Object.hashCode if given an object with the wrong type. IHasher values are optional
044 * parts of OrderedMap, OrderedSet, Arrangement, and the various classes that use Arrangement
045 * like K2 and K2V1, and allow arrays to be used as keys in those collections while keeping
046 * hashing by value instead of the normal hashing by reference for arrays. You probably won't
047 * ever need to make a class that implements IHasher yourself; for some cases you may want to
048 * look at the {@link Hashers} class for additional functions.
049 * <br>
050 * The inner classes provide alternate, faster hashing algorithms. Lightning, Wisp, and Falcon
051 * have no theoretical basis or grounds in any reason other than empirical testing for why they
052 * do what they do, and this seems to be in line with many widely-used hashes (see: The Art of
053 * Hashing, http://eternallyconfuzzled.com/tuts/algorithms/jsw_tut_hashing.aspx ). That said, Wisp
054 * performs very well, ahead of Arrays.hashCode (10.5 ms instead of 15 ms) for over a million
055 * hashes of 16-element long arrays, not including overhead for generating them, while SipHash and
056 * FNV-1a take approximately 80 ms and 135-155 ms, respectively, for the same data). Lightning and
057 * Falcon perform less-well, with Lightning taking 17 ms instead of 15 ms for Arrays.hashCode, and
058 * Falcon taking about 12.3 ms but slowing down somewhat if a 32-bit hash is requested from long
059 * data. All of these have good, low, collision rates on Strings and long arrays. Sketch is only
060 * slightly slower than Wisp, but offers little to no advantage over it yet.
061 * <br>
062 * Mist is a variant on Wisp with 128 bits for a salt-like modifier as a member variable, which can
063 * make 2 to the 128 individual hashing functions from one set of code, and uses 64 bits for some other
064 * hashes (only calls to hash() with data that doesn't involve long or double arrays). Mist has some
065 * minor resemblance to a cryptographic hash, but is not recommended it for that usage. It is,
066 * however ideal for situations that show up often in game development where end users may be able
067 * to see and possibly alter some information that you don't want changed (i.e. save data stored on
068 * a device or in the browser's LocalStorage). If you want a way to verify the data is what you
069 * think it is, you can store a hash, using one of the many-possible hash functions this can
070 * produce, somewhere else and verify that the saved data has the hash it did last time; if the
071 * exact hashing function isn't known (or exact functions aren't known) by a tampering user,
072 * then it is unlikely they can make the hash match even if they can edit it. Mist is slightly slower
073 * than Wisp, at about 18 ms for Mist for the same data instead of Wisp's 10.5, but should never be
074 * worse than twice as slow as Arrays.hashCode, and is still about three times faster than the similar
075 * SipHash that SquidLib previously had here.
076 * <br>
077 * All of the hashes used here have about the same rate of collisions on identical data
078 * (testing used Arrays.hashCode, all the hashes in here now, and the now-removed SipHash), with
079 * any fluctuation within some small margin of error. Wisp (typically via the non-nested methods
080 * in CrossHash) and Mist are the two most likely algorithms you might use here.
081 * <br>
082 * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
083 * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
084 * in Arrays.hashCode and the former default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
085 * SipHash) did approximately as well as Lightning, with no clear patterns visible (Sip has been removed from
086 * SquidLib because it needs a lot of code and is slower than all of the current hashes). The idea is from
087 * <a href="http://www.clockandflame.com/media/Goulburn06.pdf">a technical report on visual uses for hashing (PDF)</a>.
088 * <ul>
089 * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>
090 * <li>{@link CrossHash#hash(int[])}: http://i.imgur.com/x8SDqvL.png</li>
091 * <li>(Former) CrossHash.Sip.hash(int[]): http://i.imgur.com/keSpIwm.png</li>
092 * <li>{@link CrossHash.Lightning#hash(int[])}: http://i.imgur.com/afGJ9cA.png</li>
093 * </ul>
094 * <br>
095 * Note: This class was formerly called StableHash, but since that refers to a specific
096 * category of hashing algorithm that this is not, and since the goal is to be cross-
097 * platform, the name was changed to CrossHash.
098 * Note 2: FNV-1a was removed from SquidLib on July 25, 2017, and replaced as default with Wisp; Wisp
099 * was later replaced as default by Hive. Wisp was used because at the time SquidLib preferred 64-bit
100 * math when math needed to be the same across platforms; math on longs behaves the same on GWT as on
101 * desktop, despite being slower. Hive passes SMHasher, a testing suite for hashes, where Wisp does
102 * not (it fails just like Arrays.hashCode() does). Hive now uses a cross-platform subset of the
103 * possible 32-bit math operations when producing 32-bit hashes of data that doesn't involve longs or
104 * doubles, and this should speed up the default CrossHash.hash() methods a lot on GWT. 
105 * <br>
106 * Created by Tommy Ettinger on 1/16/2016.
107 * @author Tommy Ettinger
108 */

111        if (data == null)
112            return 0L;
113        long result = 0x1A976FDF6BF60B8EL, z = 0x60642E2A34326F15L;
114        final int len = data.length();
115        for (int i = 0; i < len; i++) {
116            result ^= (z += (data.charAt(i) ^ 0xC6BC279692B5CC85L) * 0x6C8E9CF570932BABL);
117            result = (result << 54 | result >>> 10);
118        }
119        result += (z ^ z >>> 26) * 0x632BE59BD9B4E019L;
120        result = (result ^ result >>> 33) * 0xFF51AFD7ED558CCDL;
121        return ((result ^ result >>> 33) * 0xC4CEB9FE1A85EC53L);

812    /**
813     * An interface that can be used to move the logic for the hashCode() and equals() methods from a class' methods to
814     * an implementation of IHasher that certain collections in SquidLib can use. Primarily useful when the key type is
815     * an array, which normally doesn't work as expected in Java hash-based collections, but can if the right collection
816     * and IHasher are used. See also {@link Hashers} for additional implementations, some of which need dependencies on
817     * things the rest of CrossHash doesn't, like a case-insensitive String hasher/equator that uses RegExodus to handle
818     * CharSequence comparison on GWT.
819     */
820    public interface IHasher extends Serializable {
821        /**
822         * If data is a type that this IHasher can specifically hash, this method should use that specific hash; in
823         * other situations, it should simply delegate to calling {@link Object#hashCode()} on data. The body of an
824         * implementation of this method can be very small; for an IHasher that is meant for byte arrays, the body could
825         * be: {@code return (data instanceof byte[]) ? CrossHash.Lightning.hash((byte[]) data) : data.hashCode();}
826         *
827         * @param data the Object to hash; this method should take any type but often has special behavior for one type
828         * @return a 32-bit int hash code of data
829         */
830        int hash(final Object data);
831
832        /**
833         * Not all types you might want to use an IHasher on meaningfully implement .equals(), such as array types; in
834         * these situations the areEqual method helps quickly check for equality by potentially having special logic for
835         * the type this is meant to check. The body of implementations for this method can be fairly small; for byte
836         * arrays, it looks like: {@code return left == right
837         * || ((left instanceof byte[] && right instanceof byte[])
838         * ? Arrays.equals((byte[]) left, (byte[]) right)
839         * : Objects.equals(left, right));} , but for multidimensional arrays you should use the
840         * {@link #equalityHelper(Object[], Object[], IHasher)} method with an IHasher for the inner arrays that are 1D
841         * or otherwise already-hash-able, as can be seen in the body of the implementation for 2D char arrays, where
842         * charHasher is an existing IHasher that handles 1D arrays:
843         * {@code return left == right
844         * || ((left instanceof char[][] && right instanceof char[][])
845         * ? equalityHelper((char[][]) left, (char[][]) right, charHasher)
846         * : Objects.equals(left, right));}
847         *
848         * @param left  allowed to be null; most implementations will have special behavior for one type
849         * @param right allowed to be null; most implementations will have special behavior for one type
850         * @return true if left is equal to right (preferably by value, but reference equality may sometimes be needed)
851         */
852        boolean areEqual(final Object left, final Object right);
853    }

855    /**
856     * Not a general-purpose method; meant to ease implementation of {@link IHasher#areEqual(Object, Object)}
857     * methods when the type being compared is a multi-dimensional array (which normally requires the heavyweight method
858     * {@link Arrays#deepEquals(Object[], Object[])} or doing more work yourself; this reduces the work needed to
859     * implement fixed-depth equality). As mentioned in the docs for {@link IHasher#areEqual(Object, Object)}, example
860     * code that hashes 2D char arrays can be done using an IHasher for 1D char arrays called charHasher:
861     * {@code return left == right
862     * || ((left instanceof char[][] && right instanceof char[][])
863     * ? equalityHelper((char[][]) left, (char[][]) right, charHasher)
864     * : Objects.equals(left, right));}
865     *
866     * @param left an array of some kind of Object, usually an array, that the given IHasher can compare
867     * @param right an array of some kind of Object, usually an array, that the given IHasher can compare
868     * @param inner an IHasher to compare items in left with items in right
869     * @return true if the contents of left and right are equal by the given IHasher, otherwise false
870     */
871    public static boolean equalityHelper(Object[] left, Object[] right, IHasher inner) {
872        if (left == right)
873            return true;
874        if (left == null || right == null || left.length != right.length)
875            return false;
876        for (int i = 0; i < left.length; i++) {
877            if (!inner.areEqual(left[i], right[i]))
878                return false;
879        }
880        return true;
881    }

1271    /**
1272     * A quick, simple hashing function that seems to have good results. Like LightRNG, it stores a state that
1273     * it updates independently of the output, and this starts at a large prime. At each step, it takes the
1274     * current item in the array being hashed, adds a large non-prime used in LightRNG's generation function
1275     * (it's 2 to the 64, times the golden ratio phi, and truncated to a signed long), multiplies by a prime
1276     * called the "state multiplier", adds the result to the state and stores it, multiplies the value of the
1277     * state by another prime called the "output multiplier", then XORs the current result with that value
1278     * before moving onto the next item in the array. A finalization step XORs the result with a complex value
1279     * made by adding the state (left over from the previous step) to what was the output multiplier, adding
1280     * the last known value for result to the phi-related constant from LightRNG, multiplying that pair, adding
1281     * the initial state (which turns out to be unusually good for this, despite no particularly special numeric
1282     * qualities other than being a probable prime) and then bitwise-rotating it left by a seemingly-random
1283     * number drawn from the highest 6 bits of the state.
1284     * <br>
1285     * This all can be done very quickly; a million hashes of a million different 16-element long arrays can be
1286     * computed in under 18-20 ms (in the benchmark, some amount of that is overhead from generating a new
1287     * array with LongPeriodRNG, since the benchmark uses that RNG's state for data, and the default
1288     * Arrays.hashCode implementation is only somewhat faster at under 16 ms). After several tries and tweaks
1289     * to the constants this uses, it also gets remarkably few hash collisions. On the same 0x100000, or
1290     * 1048576, RNG states for data, Lightning gets 110 collisions, the JDK Arrays.hashCode method gets 129
1291     * collisions, Sip (implementing SipHash) gets 145 collisions, and CrossHash (using the FNV-1a algorithm)
1292     * gets 135 collisions. Dispersion is not perfect, but
1293     * at many bit sizes Lightning continues to have less collisions (it disperses better than the other hashes
1294     * with several quantities of bits, at least on this test data). Lightning also does relatively well, though
1295     * it isn't clearly ahead of the rest all the time, when hashing Strings, especially ones that use a larger
1296     * set of letters, it seems (FakeLanguageGen was used to make test data, and languages that used more
1297     * characters in their alphabet seemed to hash better with this than competing hashes for some reason).
1298     * <br>
1299     * There is certainly room for improvement with the specific numbers chosen; earlier versions used the
1300     * state multiplier "Neely's number", which is a probable prime made by taking the radix-29 number
1301     * "HARGNALLINSCLOPIOPEEPIO" (a reference to a very unusual TV show), truncating to 64 bits, and rotating
1302     * right by 42 bits. This version uses "Neely's number" for an initial state and during finalization, and
1303     * uses a different probable prime as the state multiplier, made with a similar process; it starts with the
1304     * radix-36 number "EDSGERWDIJKSTRA", then does the same process but rotates right by 37 bits to obtain a
1305     * different prime. This tweak seems to help with hash collisions. Extensive trial and error was used to
1306     * find the current output multiplier, which has no real relationship to anything else but has exactly 32 of
1307     * 64 bits set to 1, has 1 in the least and most significant bit indices (meaning it is negative and odd),
1308     * and other than that seems to have better results on most inputs for mystifying reasons. Earlier versions
1309     * applied a Gray code step to alter the output instead of a multiplier that heavily overflows to obfuscate
1310     * state, but that had a predictable pattern for most of the inputs tried, which seemed less-than-ideal for
1311     * a hash. Vitally, Lightning avoids predictable collisions that Arrays.hashCode has, like
1312     * {@code Arrays.hashCode(new long[]{0})==Arrays.hashCode(new long[]{-1})}.
1313     * <br>
1314     * The output multiplier is 0xC6BC279692B5CC83L, the state multiplier is 0xD0E89D2D311E289FL, the number
1315     * added to the state (from LightRNG and code derived from FastUtil, but obtained from the golden ratio
1316     * phi) is 0x9E3779B97F4A7C15L, and the starting state ("Neely's Number") is 0x632BE59BD9B4E019L.
1317     * <br>
1318     * To help find patterns in hash output in a visual way, you can hash an x,y point, take the bottom 24 bits,
1319     * and use that as an RGB color for the pixel at that x,y point. On a 512x512 grid of points, the patterns
1320     * in Arrays.hashCode and the default CrossHash algorithm (FNV-1a) are evident, and Sip (implementing
1321     * SipHash) does approximately as well as Lightning, with no clear patterns visible (Sip has been removed
1322     * from SquidLib because it needs a lot of code and is slower than Mist). The
1323     * idea is from a technical report on visual uses for hashing,
1324     * http://www.clockandflame.com/media/Goulburn06.pdf .
1325     * <ul>
1326     * <li>{@link java.util.Arrays#hashCode(int[])}: http://i.imgur.com/S4Gh1sX.png</li>

4154    /**
4155     * A reasonably-fast hashing function that passes SMHasher's quality tests even on {@code long[]} inputs (passing on
4156     * those should mean it can pass with other types of array as well). This is the default used by methods in the
4157     * outer class like {@link CrossHash#hash(int[])}, which should be the same as {@link CrossHash.Hive#hash(int[])};
4158     * you should prefer using the outer class since this inner class is only here for reference.
4159     * <br>
4160     * This mixes three different algorithms: the main one is used whenever inputs or outputs are 64-bit (so, all hash64()
4161     * overloads and {@link Hive#hash(long[])} for long and double), a modification of the main one to perform better on
4162     * GWT (only used on hash() overloads, and only when inputs are individually 32-bit or less), and
4163     * {@link CrossHash.Jolt}'s algorithm for hash64() on boolean and byte data.
4164     * <br>
4165     * Speed-wise, the main algorithm is about 20% slower than Wisp, but in hash tables it doesn't have clea failure 
4166     * cases like Wisp does on some inputs (such as fixed-length Strings with identical prefixes). If collisions are
4167     * expensive or profiling shows that Wisp's algorithm is colliding at a high rate, you should probably use the
4168     * normal IHasher and CrossHash.hash() methods, since those will use Hive. The modified algorithm for GWT is a
4169     * little slower than the main algorithm in the C++ implementation that was used to check SMHasher quality, but it

4172     * represented by 3 JS numbers, so barring special optimizations it takes at least 3x as many math operations to use
