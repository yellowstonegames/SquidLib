012/**
013 * Region encoding of on/off information about areas using bitsets; uncompressed (fatty), but fast (greased lightning).
014 * This can handle any size of 2D data, and is not strictly limited to 256x256 as CoordPacker is. It stores several long
015 * arrays and uses each bit in one of those numbers to represent a single point, though sometimes this does waste bits
016 * if the height of the area this encodes is not a multiple of 64 (if you store a 80x64 map, this uses 80 longs; if you
017 * store an 80x65 map, this uses 160 longs, 80 for the first 64 rows and 80 more to store the next row). It's much
018 * faster than CoordPacker at certain operations (anything that expands or retracts an area, including
019 * {@link #expand()}), {@link #retract()}), {@link #fringe()}), {@link #surface()}, and {@link #flood(GreasedRegion)},
020 * and slightly faster on others, like {@link #and(GreasedRegion)} (called intersectPacked() in CoordPacker) and
021 * {@link #or(GreasedRegion)} (called unionPacked() in CoordPacker).
022 * <br>
023 * Each GreasedRegion is mutable, and instance methods typically modify that instance and return it for chaining. There
024 * are exceptions, usually where multiple GreasedRegion values are returned and the instance is not modified.
025 * <br>
026 * Typical usage involves constructing a GreasedRegion from some input data, like a char[][] for a map or a double[][]
027 * from DijkstraMap, and modifying it spatially with expand(), retract(), flood(), etc. It's common to mix in data from
028 * other GreasedRegions with and() (which gets the intersection of two GreasedRegions and stores it in one), or() (which
029 * is like and() but for the union), xor() (like and() but for exclusive or, finding only cells that are on in exactly
030 * one of the two GreasedRegions), and andNot() (which can be considered the "subtract another region from me" method).
031 * There are 8-way (Chebyshev distance) variants on all of the spatial methods, and methods without "8way" in the name
032 * are either 4-way (Manhattan distance) or not affected by distance measurement. Once you have a GreasedRegion, you may
033 * want to get a single random point from it (use {@link #singleRandom(RNG)}), get several random points from it (use
034 * {@link #randomPortion(RNG, int)} for random sampling or {@link #randomSeparated(double, RNG)} for points that have
035 * some distance between each other), or get all points from it (use {@link #asCoords()}. You may also want to produce
036 * some 2D data from one or more GreasedRegions, such as with {@link #sum(GreasedRegion...)} or {@link #toChars()}. The
037 * most effective techniques regarding GreasedRegion involve multiple methods, like getting a few random points from an
038 * existing GreasedRegion representing floor tiles in a dungeon with {@link #randomPortion(RNG, int)}, then inserting
039 * those into a new GreasedRegion with {@link #insertSeveral(Coord...)}, and then finding a random expansion of those
040 * initial points with {@link #spill(GreasedRegion, int, RNG)}, giving the original GreasedRegion of floor tiles as the
041 * first argument. This could be used to position puddles of water or patches of mold in a dungeon level, while still
042 * keeping the starting points and finished points within the boundaries of valid (floor) cells.
043 * <br>
044 * For efficiency, you can place one GreasedRegion into another (typically a temporary value that is no longer needed
045 * and can be recycled) using {@link #remake(GreasedRegion)}, or give the information that would normally be used to
046 * construct a fresh GreasedRegion to an existing one of the same dimensions with {@link #refill(boolean[][])} or any
047 * of the overloads of refill(). These re-methods don't do as much work as a constructor does if the width and height
048 * of their argument are identical to their current width and height, and don't create more garbage for the GC.
049 * <br>
050 * Created by Tommy Ettinger on 6/24/2016.
051 */

072    public GreasedRegion(boolean[][] bits)
073    {
074        width = bits.length;
075        height = bits[0].length;
076        ySections = (height + 63) >> 6;
077        yEndMask = -1L >>> (64 - (height & 63));
078        data = new long[width * ySections];
079        for (int x = 0; x < width; x++) {
080            for (int y = 0; y < height; y++) {
081                if(bits[x][y]) data[x * ySections + (y >> 6)] |= 1L << (y & 63);
082            }
083        }

520        if (width == other.width && height == other.height) {
521            System.arraycopy(other.data, 0, data, 0, width * ySections);

1675     * ArrayList of GreasedRegion, with one GreasedRegion per isolated area. Useful when you have, for example, all the
1676     * rooms in a dungeon with their connecting corridors removed, but want to separate the rooms. You can get the
1677     * aforementioned data assuming a bare dungeon called map using:
1678     * <br>
1679     * {@code GreasedRegion floors = new GreasedRegion(map, '.'),
1680     * rooms = floors.copy().retract8way().flood(floors, 2),
1681     * corridors = floors.copy().andNot(rooms),
1682     * doors = rooms.copy().and(corridors.copy().fringe());}
1683     * <br>
1684     * You can then get all rooms as separate regions with {@code List<GreasedRegion> apart = split(rooms);}, or
1685     * substitute {@code split(corridors)} to get the corridors. The room-finding technique works by shrinking floors
1686     * by a radius of 1 (8-way), which causes thin areas like corridors of 2 or less width to be removed, then
1687     * flood-filling the floors out from the area that produces by 2 cells (4-way this time) to restore the original
1688     * size of non-corridor areas (plus some extra to ensure odd shapes are kept). Corridors are obtained by removing
1689     * the rooms from floors. The example code also gets the doors (which overlap with rooms, not corridors) by finding
1690     * where the a room and a corridor are adjacent. This technique is used with some enhancements in the RoomFinder
1691     * class.
1692     * @see squidpony.squidgrid.mapping.RoomFinder for a class that uses this technique without exposing GreasedRegion
1693     * @return an ArrayList containing each unconnected area from packed as a GreasedRegion element
1694     */

1703            remaining.andNot(filled);
1704            fst = remaining.first();
1705        }
1706        return scattered;
