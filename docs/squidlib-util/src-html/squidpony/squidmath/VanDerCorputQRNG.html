007/**
008 * A quasi-random number generator that goes through one of many sub-random sequences found by J.G. van der Corput.
009 * More specifically, this offers both the normal van der Corput sequence, which only changes the state by incrementing
010 * it (this works better in the normal usage as part of a 2D or 3D point generator) and a kind of scrambled van der
011 * Corput sequence, where the state changes unpredictably (this works better when using this to generate 1D sequences,
012 * or when the base may be larger than 7 or non-prime). In both cases, the state is internally stored in a 64-bit long
013 * that is incremented once per generated number, but when scramble is true, the state is altered with something similar
014 * to a Gray code before being used; more on this later if you want to read about it. The important things to know about
015 * this class are: size of state affects speed (prefer smaller seeds, but quality is sometimes a bit poor at first if
016 * you start at 0); the base (when given) should be prime and moderately small (or very small if scramble is false, any
017 * of 2, 3, 5, or 7 should be safe); this doesn't generate very random numbers when scramble is false (which can be good
018 * for making points that should not overlap), but it will seem much more random when scramble is true; this is a
019 * StatefulRandomness with an additional method for generating quasi-random doubles, {@link #nextDouble()}; and there
020 * are several static methods offered for convenient generation of points on the related Halton sequence (as well as
021 * faster generation of doubles in the base-2 van der Corput sequence, and a special method that switches which base
022 * it uses depending on the index to seem even less clearly-patterned).
023 * <br>
024 * This generator allows a base (also called a radix) that changes the sequence significantly; a base should be prime,
025 * and this performs a little better in terms of time used with larger primes, though quality is also improved by
026 * preferring primes that aren't very large relative to the quantity of numbers expected to be generated. Unfortunately,
027 * performance is not especially similar to conventional PRNGs; smaller (positive) state values are processed more
028 * quickly than larger ones, or most negative states. At least one 64-bit integer modulus and one 64-bit integer
029 * division are required for any number to be produced, and the amount of both of these relatively-non-cheap operations
030 * increases linearly with the number of digits (in the specified base, which is usually not 10) needed to represent
031 * the current state. Since performance is not optimal, and the results are rather strange anyway relative to PRNGs,
032 * this should not be used as a direct substitute for a typical RandomnessSource (however, it is similar to, but simpler
033 * than, {@link squidpony.squidmath.SobolQRNG}). So what's it good for?
034 * <br>
035 * A VanDerCorputSequence can be a nice building block for more complicated quasi-randomness, especially for points in
036 * 2D or 3D, when scramble is false. There's a simple way that should almost always "just work" as the static method
037 * {@link #halton(int, int, int)} here. If it doesn't meet your needs, there's a little more complexity involved. Using
038 * a VanDerCorputQRNG with base 3 for the x-axis and another VanDerCorputQRNG with base 5 for the y-axis, requesting a
039 * double from each to make points between (0.0, 0.0) and (1.0, 1.0), has an interesting trait that can be desirable for
040 * many kinds of positioning in 2D: once a point has been generated, an identical point will never be generated until
041 * floating-point precision runs out, but more than that, nearby points will almost never be generated for many
042 * generations, and most points will stay at a comfortable distance from each other if the bases are different and both
043 * prime (as well as, more technically, if they share no common denominators). This is also known as a Halton sequence,
044 * which is a group of sequences of points instead of simply numbers. The choices of 3 and 5 are just examples; any two
045 * different primes will technically work for 2D, but patterns can become noticeable with primes larger than about 7,
046 * with 11 and 13 sometimes acceptable. Three VanDerCorputQRNG sequences could be used for a Halton sequence of 3D
047 * points, using three different prime bases, four for 4D, etc. SobolQRNG can be used for the same purpose, but the
048 * points it generates are typically more closely-aligned to a specific pattern, the pattern is symmetrical between all
049 * four quadrants of the square between (0.0, 0.0) and (1.0, 1.0) in 2D, and it probably extends into higher dimensions.
050 * Using one of the possible Halton sequences gives some more flexibility in the kinds of random-like points produced.
051 * <br>
052 * Because just using the state in a simple incremental fashion puts some difficult requirements on the choice of base
053 * and seed, we can use a technique like Gray codes to scramble the state. The sequence of these Gray-like codes for the
054 * integers from 0 to 16 is {@code 0, 9, 21, 16, 50, 51, 23, 10, 4, 28, 49, 39, 246, 198, 179, 97, 393}, and this can be
055 * generated efficiently with {@code (i * i) ^ ((i * 137) >>> 4)}. No duplicate results were found in any numbers of 20
056 * bits or less, and trying to find duplicates in 24 bits exhausted the testing computer's memory, without finding a
057 * duplicate. You should be fine.
058 * <br>
059 * Expected output for {@link #nextDouble()} called 33 times on an instance made with {@code new VanDerCorputQRNG(11, 83L, true)}:
060 * 0.5194317328051362, 0.8590943241581859, 0.5931288846390274, 0.5045420394781778, 0.7892220476743392,
061 * 0.6645037907246772, 0.7809575848644218, 0.0725360289597705, 0.6322655556314459, 0.6998838877125879,
062 * 0.5578853903421898, 0.5969537599890717, 0.6323338569769824, 0.873505908066389, 0.09514377433235435,
063 * 0.6623864490130456, 0.4376750221979373, 0.6946929854518133, 0.4250392732736835, 0.5294720305990028,
064 * 0.4790656375930606, 0.2626869749334062, 0.056075404685472306, 0.9758213236800765, 0.676729731575712,
065 * 0.899118912642579, 0.1876237961887849, 0.652755959292398, 0.039683081756710606, 0.10504746943514787,
066 * 0.2598183184208729, 0.4078273341984837, 0.3034628782187009
067 * <br>
068 * Expected output for {@link #nextDouble()} called 33 times on an instance made with {@code new VanDerCorputQRNG(19, 83L, true)}:
069 * 0.8944452544102639, 0.7842327790609341, 0.4352023081468067, 0.0696971324652205, 0.5957213342439054,
070 * 0.8864342661581787, 0.0167739658228528, 0.8956192785506557, 0.26943470353972115, 0.3525371966145134,
071 * 0.09754375733765087, 0.8924118139056637, 0.3585147443619984, 0.9126771587081131, 0.28926266679967155,
072 * 0.8542138258607592, 0.8789987799356972, 0.7905019145034184, 0.1624220194749887, 0.9349836173755572,
073 * 0.48623015477167914, 0.6716799287912155, 0.8240344994283346, 0.6557883994137552, 0.4561966221867542,
074 * 0.07946532024769608, 0.15134168706501638, 0.1382202407900492, 0.7890439760284221, 0.7142517322611092,
075 * 0.3069037223471275, 0.7030256060036372, 0.01678163918324752
076 * <br>
077 * Note on Gray-like code implementation: This is not a typical Gray code. Normally, the operation looks like
078 * {@code i ^ (i >>> 1)}, which gives negative results for negative values of i (not wanted here), and also clusters
079 * two very similar numbers together for every pair of sequential numbers. An earlier version scrambled with a basic
080 * Gray code, but the current style, {@code (i * i) ^ ((i * 137) >>> 4)}, produces much more "wild and crazy" results,
081 * but never negative ones. The period of this, if it is seen as a RandomnessSource, is probably about 2^56, at least
082 * for reasonably small starting states (under 1000 or so?). This is ideal for a scramble.
083 * <br>
084 * Created by Tommy Ettinger on 11/18/2016.
085 */
086public class VanDerCorputQRNG implements StatefulRandomness, RandomnessSource, Serializable {
087    private static final long serialVersionUID = 5;
088    public long state;
089    public final int base;
090    public final boolean scramble;
091    /**
092     * Constructs a new van der Corput sequence generator with base 13, starting point 83, and scrambling enabled.
093     */
094    public VanDerCorputQRNG()
095    {
096        base = 7;
097        state = 37;
098        scramble = true;
099    }
100
101    /**
102     * Constructs a new van der Corput sequence generator with the given starting point in the sequence as a seed.
103     * Usually seed should be at least 20 with this constructor, but not drastically larger; 2000 is probably too much.
104     * This will use a base 13 van der Corput sequence and have scrambling enabled.
105     * @param seed the seed as a long that will be used as the starting point in the sequence; ideally positive but low
106     */
107    public VanDerCorputQRNG(long seed)
108    {
109        base = 7;
110        state = seed;
111        scramble = true;
112    }
113
114    /**
115     * Constructs a new van der Corput sequence generator with the given base (a.k.a. radix; if given a base less than
116     * 2, this will use base 2 instead) and starting point in the sequence as a seed. Good choices for base are between
117     * 10 and 60 or so, and should usually be prime. Good choices for seed are larger than base but not by very much,
118     * and should generally be positive at construction time.
119     * @param base the base or radix used for this VanDerCorputQRNG; for most uses this should be prime but small-ish
120     * @param seed the seed as a long that will be used as the starting point in the sequence; ideally positive but low
121     * @param scramble if true, will produce more-random values that are better for 1D; if false, better for 2D or 3D
122     */
123    public VanDerCorputQRNG(int base, long seed, boolean scramble)
124    {
125        this.base = base < 2 ? 2 : base;
126        state = seed;
127        this.scramble = scramble;
128    }
129
130    /**
131     * Gets the next quasi-random long as a fraction of {@link Long#MAX_VALUE}; this can never produce a negative value.
132     * It is extremely unlikely to produce two identical values unless the state is very high or is negative; state
133     * increases by exactly 1 each time this, {@link #next(int)}, or {@link #nextDouble()} is called and can potentially
134     * wrap around to negative values after many generations.
135     * @return a quasi-random non-negative long; may return 0 rarely, probably can't return {@link Long#MAX_VALUE}
136     */
137    @Override
138    public long nextLong() {
139        // when scrambling the sequence, intentionally uses a non-standard Gray-like code
140        long s = (scramble) ? (++state & 0x7fffffffffffffffL) : (++state * state) ^ (state * 137 >>> 4),
141                num = s % base, den = base;
142        while (den <= s) {
143            num *= base;
144            num += (s % (den * base)) / den;
145            den *= base;
146        }
147        return (Long.MAX_VALUE / den) * num;
148    }
149
150    @Override
151    public int next(int bits) {
152        return (int)(nextLong()) >>> (32 - bits);
153    }
154
155    /**
156     * Gets the next quasi-random double from between 0.0 and 1.0 (normally both exclusive; only if state is negative or
157     * has wrapped around to a negative value can 0.0 ever be produced). It should be nearly impossible for this to
158     * return the same number twice unless floating-point precision has been exhausted or a very large amount of numbers
159     * have already been generated. Certain unusual bases may make this more likely, and similar numbers may be returned
160     * more frequently if scramble is true.
161     * @return a quasi-random double that will always be less than 1.0 and will be no lower than 0.0
162     */
163    public double nextDouble() {
164        // when scrambling the sequence, intentionally uses a non-standard Gray-like code
165        long s = (scramble) ? (++state & 0x7fffffffffffffffL) : (++state * state) ^ (state * 137 >>> 4),
166                num = s % base, den = base;
167        while (den <= s) {
168            num *= base;
169            num += (s % (den * base)) / den;
170            den *= base;
171        }
172        return num / (double)den;
173    }
174
175    @Override
176    public VanDerCorputQRNG copy() {
177        return new VanDerCorputQRNG(base, state, scramble);
178    }
179
180    @Override
181    public long getState() {
182        return state;
183    }
184
185    @Override
186    public void setState(long state) {
187        this.state = state;
188    }
189
190    @Override
191    public String toString() {
192        return "VanDerCorputQRNG with base " + base +
193                ", scrambling " + (scramble ? "on" : "off") +
194                ", and state 0x" + StringKit.hex(state);
195    }
196
197    @Override
198    public boolean equals(Object o) {
199        if (this == o) return true;
200        if (o == null || getClass() != o.getClass()) return false;
201
202        VanDerCorputQRNG that = (VanDerCorputQRNG) o;
203
204        return state == that.state && base == that.base && scramble == that.scramble;
205
206    }
207
208    @Override
209    public int hashCode() {
210        int result = (int) (state ^ (state >>> 32));
211        result = 31 * result + base;
212        result = 31 * result + (scramble ? 1 : 0);
213        return result;
214    }

216    /**
217     * Convenience method that gets a quasi-random Coord between integer (0,0) inclusive and (width,height) exclusive.
218     * This is roughly equivalent to creating two VanDerCorputQRNG generators, one with
219     * {@code new VanDerCorputQRNG(2, index, false)} and the other with {@code new VanDerCorputQRNG(3, index, false)},
220     * then getting an x-coordinate from the first with {@code (int)(nextDouble() * width)} and similarly for y with the
221     * other generator. The advantage here is you don't actually create any objects using this static method, other than
222     * a (almost always shared) reference to the returned Coord. You might find an advantage in using values for index
223     * that start higher than 20 or so, but you can pass sequential values for index and generally get Coords that won't
224     * be near each other; this is not true for all parameters to Halton sequences, but it is true for this one.
225     * @param width the maximum exclusive bound for the x-positions of Coord values this can return
226     * @param height the maximum exclusive bound for the y-positions of Coord values this can return
227     * @param index an int that, if unique, positive, and not too large, will usually result in unique Coord values
228     * @return a Coord that usually will have a comfortable distance from Coords produced with close index values
229     */
230    public static Coord halton(int width, int height, int index)
231    {
232        int s = (index+1 & 0x7fffffff),
233                numY = s % 3, denY = 3;
234        while (denY <= s) {
235            numY *= 3;
236            numY += (s % (denY * 3)) / denY;
237            denY *= 3;
238        }
239        return Coord.get((int)(width * determine2(s)), numY * height / denY);
240    }
241    /**
242     * Convenience method that gets a quasi-random Coord3D between integer (0,0,0) inclusive and (width,height,depth)
243     * exclusive. This is roughly equivalent to creating three VanDerCorputQRNG generators, one with
244     * {@code new VanDerCorputQRNG(2, index, false)} another with {@code new VanDerCorputQRNG(3, index, false)},
245     * and another with {@code new VanDerCorputQRNG(5, index, false)}, then getting an x-coordinate from the first with
246     * {@code (int)(nextDouble() * width)} and similarly for y and z with the other generators. The advantage here is
247     * you don't actually create any objects using this static method, other than a returned Coord3D. You might find an
248     * advantage in using values for index that start higher than 20 or so, but you can pass sequential values for index
249     * and generally get Coord3Ds that won't be near each other; this is not true for all parameters to Halton
250     * sequences, but it is true for this one.
251     * @param width the maximum exclusive bound for the x-positions of Coord3D values this can return
252     * @param height the maximum exclusive bound for the y-positions of Coord3D values this can return
253     * @param depth the maximum exclusive bound for the z-positions of Coord3D values this can return
254     * @param index an int that, if unique, positive, and not too large, will usually result in unique Coord3D values
255     * @return a Coord3D that usually will have a comfortable distance from Coord3Ds produced with close index values
256     */
257    public static Coord3D halton(int width, int height, int depth, int index)
258    {
259        int s = (index+1 & 0x7fffffff),
260                numY = s % 3, denY = 3, numZ = s % 5, denZ = 5;
261        while (denY <= s) {
262            numY *= 3;
263            numY += (s % (denY * 3)) / denY;
264            denY *= 3;
265        }
266        while (denY <= s) {
267            numZ *= 5;
268            numZ += (s % (denZ * 5)) / denZ;
269            denZ *= 5;
270        }
271        return Coord3D.get((int)(width * determine2(s)), numY * height / denY, numZ * depth / denZ);
272    }

274    /**
275     * Convenience method to get a double from the van der Corput sequence with the given {@code base} at the requested
276     * {@code index} without needing to construct a VanDerCorputQRNG. You should use a prime number for base; 2, 3, 5,
277     * and 7 should be among the first choices. This does not perform any scrambling on index other than incrementing it
278     * and ensuring it is positive (by discarding the sign bit; for all positive index values other than 0x7FFFFFFF,
279     * this has no effect).
280     * <br>
281     * Delegates to {@link #determine2(int)} when base is 2, which should offer some speed improvement.
282     * @param base a (typically very small) prime number to use as the base/radix of the van der Corput sequence
283     * @param index the position in the sequence of the requested base
284     * @return a quasi-random double between 0.0 (inclusive) and 1.0 (exclusive).
285     */
286    public static double determine(int base, int index)
287    {
288        if(base == 2)
289            return determine2(index);
290        int s = (index+1 & 0x7fffffff),
291                num = s % base, den = base;
292        while (den <= s) {
293            num *= base;
294            num += (s % (den * base)) / den;
295            den *= base;
296        }
297        return num / (double)den;
298    }
299    /**
300     * Convenience method to get a double from the van der Corput sequence with the base 2 at the requested
301     * {@code index} without needing to construct a VanDerCorputQRNG. This does not perform any scrambling on index
302     * other than incrementing it and ensuring it is positive (by discarding the sign bit; for all positive index values
303     * other than 0x7FFFFFFF, this has no effect).
304     * <br>
305     * Because binary manipulation of numbers is easier and more efficient, this method should be somewhat faster than
306     * the alternatives, like {@link #determine(int, int)} with base 2.
307     * @param index the position in the sequence of the requested base
308     * @return a quasi-random double between 0.0 (inclusive) and 1.0 (exclusive).
309     */
310    public static double determine2(int index)
311    {
312        int s = (index+1 & 0x7fffffff), leading = Integer.numberOfLeadingZeros(s);
313        return (Integer.reverse(s) >>> leading) / (double)(1 << (32 - leading));
314    }
315
316    private static final int[] lowPrimes = {2, 3, 2, 3, 5, 2, 3, 2};
317
318    /**
319     * Chooses one sequence from the van der Corput sequences with bases 2, 3, and 5, where 5 is used 1/8 of the time,
320     * 3 is used 3/8 of the time, and 2 is used 1/2 of the time, and returns a double from the chosen sequence at the
321     * specified {@code index}. The exact setup used for the choice this makes is potentially fragile, but in certain
322     * circumstances this does better than {@link SobolQRNG} at avoiding extremely close values (the kind that overlap
323     * on actual maps). Speed is not a concern here; this should be very much fast enough for the expected usage in
324     * map generation (it's used in {@link GreasedRegion#quasiRandomSeparated(double)}.
325     * @param index the index to use from one of the sequences
326     * @return a double from 0.0 (inclusive, but extremely rare) to 1.0 (exclusive); values will tend to spread apart
327     */
328    public static double determineMixed(int index)
329    {
330        return determine(lowPrimes[index & 7], index);
331    }
